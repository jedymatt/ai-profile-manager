# aipm Installer — Design Spec

**Date:** 2026-06-28
**Status:** Approved design, pre-implementation
**Scope:** Add a prebuilt-binary installer for `aipm`, generated and maintained by `dist`.

## Summary

Today `aipm` can only be installed by building from source (`cargo install --path .`
or `cargo build --release`), which requires a Rust toolchain. This spec adds a
**prebuilt-binary installer**: a one-line `curl … | sh` (and PowerShell) command
that downloads a matching `aipm` binary from GitHub Releases and places it on the
user's PATH — **no Rust toolchain required**.

We adopt [`dist`](https://github.com/axodotdev/cargo-dist) (formerly cargo-dist),
the standard tool for distributing Rust binaries. From config in `Cargo.toml`,
`dist` generates both the installer scripts (`install.sh`, `install.ps1`) and a
GitHub Actions release workflow that builds per-target binaries and publishes them,
with checksums, to a GitHub Release.

## Goals

- Install `aipm` without a Rust toolchain, via a single copy-paste command.
- Cover the platforms `aipm` already supports: Linux, macOS, Windows.
- Keep release infrastructure generated and maintainable (not hand-rolled), so
  target detection, checksums, and archive handling are not our responsibility.
- Keep testing and releasing decoupled: the existing test CI is untouched; release
  is a separate, tag-triggered workflow.
- Default install location works for users who have **no** cargo (`~/.local/bin`).

## Non-goals (this iteration)

- **Homebrew tap, npm installer, MSI/winget.** Easy to add later via `dist` config.
- **Linux ARM (aarch64) and musl/static targets.** Deferred; the "Standard" target
  set is chosen for coverage vs. build simplicity.
- **Automated version bumping / release-please.** Releases are cut manually by
  tagging.

## Decisions (from brainstorming)

| Decision        | Choice                                                                 |
| --------------- | --------------------------------------------------------------------- |
| Installer type  | Download prebuilt binary from GitHub Releases                          |
| Mechanism       | `dist` (generates installers + release CI from `Cargo.toml`)           |
| Targets         | Standard: `x86_64-unknown-linux-gnu`, `x86_64-apple-darwin`, `aarch64-apple-darwin`, `x86_64-pc-windows-msvc` |
| Installers      | `shell` + `powershell` (Windows is supported)                         |
| Install path    | `~/.local/bin` (works without a Rust toolchain)                        |

## Components

### 1. `Cargo.toml` — package metadata (required by `dist`)

Add to `[package]`:

- `description` — one-line crate summary
- `repository = "https://github.com/jedymatt/ai-profile-manager"`
- `license = "MIT"` (matches `LICENSE`)
- `readme = "README.md"`

These are prerequisites for `dist` (it errors without a repository URL) and are
good crate hygiene independent of the installer.

### 2. `Cargo.toml` — `[workspace.metadata.dist]` config

```toml
[workspace.metadata.dist]
cargo-dist-version = "<pinned latest version installed locally>"
ci = ["github"]
installers = ["shell", "powershell"]
targets = [
  "x86_64-unknown-linux-gnu",
  "x86_64-apple-darwin",
  "aarch64-apple-darwin",
  "x86_64-pc-windows-msvc",
]
install-path = "~/.local/bin"
```

Plus the `[profile.dist]` build profile that `dist init` appends (an inherits-release
profile used for shippable builds).

### 3. `.github/workflows/release.yml` (generated by `dist`)

Triggered on `v*` tag push. Stages: **plan** → **build** each target on its native
runner (ubuntu/macos/windows) → **publish** binaries + checksums + generated
`install.sh`/`install.ps1` to a GitHub Release. This file is generated; it must not
be hand-edited (regenerated via `dist init` / `dist generate`).

### 4. `README.md` — Install section

Promote the prebuilt one-liners to the **primary** install path; keep `cargo install`
/ `cargo build` as the from-source alternative. Add the release procedure (tag must
match the `Cargo.toml` version) to the Development section.

## Data flow

**Release (maintainer):**
bump `version` in `Cargo.toml` → `git tag vX.Y.Z` → `git push --tags` →
`release.yml` builds all targets and publishes the GitHub Release with artifacts.

**Install (user):**
runs the one-liner → script detects OS/arch → downloads the matching archive and
verifies its checksum → extracts `aipm` into `~/.local/bin` → writes an env script
and guides the user to add the dir to PATH (installers cannot mutate a running
shell's PATH).

## Error handling / edge cases

- **Unsupported platform:** `dist` scripts exit with a clear error when no target
  matches the host.
- **Checksum mismatch:** download is aborted by the generated script.
- **`~/.local/bin` not on PATH:** `dist` writes an env file and prints guidance;
  it conservatively avoids editing arbitrary shell rc files.
- **Tag/version mismatch:** `dist` errors during planning if the pushed tag does
  not match the `Cargo.toml` version. Mitigation: documented release steps keep
  them in sync.

## Testing / verification

- `dist plan` runs clean locally (validates config and target set).
- `dist generate --check` shows no drift between config and the committed
  `release.yml`.
- Existing `cargo test --all`, `cargo clippy --all-targets -- -D warnings`, and
  `cargo fmt --all -- --check` remain green (we only add metadata + a profile, no
  source changes).
- **Manual post-merge smoke test (cannot be exercised pre-release):** cut a real
  tag, confirm the Release publishes artifacts, then run each installer one-liner
  on a clean machine/container. Tracked as a follow-up, consistent with the
  README's existing "Caveats" stance on behaviors that need a live environment.

## Dependencies / prerequisites

- `dist` must be installed locally to run `dist init` / `dist generate`. It is
  installed in the working environment at implementation time (via its own
  installer); it is **not** vendored into the repo.

## Out of scope

Homebrew/npm/MSI installers, Linux ARM and musl/static targets, and automated
version management — all addable later through `dist` config without redesign.
